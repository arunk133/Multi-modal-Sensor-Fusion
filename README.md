
# Multi-Modal Sensor Fusion Architecture (CARLA)

A real-time autonomous driving perception system implementing **multi-modal sensor fusion**, **object tracking**, and **tracking performance evaluation** inside the **CARLA Autonomous Driving Simulator**. The project integrates RGB Camera, LiDAR, and Radar data using an object-level late fusion architecture with an Extended Kalman Filter (EKF) for robust multi-object tracking.

---

# 🚗 Introduction

Autonomous vehicles rely on multiple complementary sensors to perceive their surroundings. Cameras provide semantic understanding, LiDAR offers precise 3D geometry, and Radar measures object velocity under adverse environmental conditions. Individually, each sensor has limitations. Sensor fusion combines their strengths to create a more reliable perception system.

This project implements a complete real-time sensor fusion pipeline capable of:

- Detecting road users
- Associating detections across multiple sensors
- Tracking moving vehicles
- Estimating object motion
- Evaluating tracking performance using simulation ground truth

The complete pipeline is executed inside the CARLA simulator, allowing safe testing under reproducible driving scenarios.

---

# 🚗 About CARLA Simulator

**CARLA (Car Learning to Act)** is an open-source autonomous driving simulator developed using Unreal Engine. It provides realistic urban environments for developing, training, and validating autonomous vehicle algorithms.

CARLA offers:

- Physically accurate RGB cameras
- Ray-casting LiDAR sensors
- Doppler Radar simulation
- GPS and IMU sensors
- Traffic simulation
- Dynamic weather
- Day/night cycles
- Pedestrian simulation
- Ground-truth annotations
- Synchronous simulation mode

These features make CARLA an ideal platform for testing perception and tracking algorithms before deployment on real vehicles.

# 📷 Camera Perception Module

The RGB camera performs object detection using **YOLOv8m**.

Responsibilities include:

- Vehicle detection
- Bounding box estimation
- Object classification
- Feature extraction
- Appearance descriptor generation

For every detected vehicle, a normalized color histogram is computed for later appearance matching.

Outputs include:

- Bounding box
- Confidence score
- Class label
- Image centroid
- Color histogram

---

# 🌐 LiDAR Perception Module

The LiDAR module processes raw point clouds generated by CARLA.

Processing steps include:

1. Ground filtering
2. Height thresholding
3. Region-of-interest filtering
4. DBSCAN clustering
5. Cluster centroid estimation

Each cluster corresponds to a candidate vehicle.

Outputs include:

- 3D centroid
- Cluster dimensions
- Number of points
- Object position

---

# 📡 Radar Processing Module

The Radar sensor measures:

- Relative velocity
- Distance
- Azimuth
- Elevation

Radar provides accurate velocity measurements even under poor visibility.

Outputs include:

- Target range
- Relative velocity
- Radar position

# 🔗 Multi-Sensor Fusion

The project implements **Object-Level Late Fusion**.

Instead of merging raw sensor measurements, each sensor independently detects objects.

Detected objects are then associated using:

- Euclidean distance
- Mahalanobis distance
- Appearance similarity

Advantages include:

- Modular architecture
- Lower computational complexity
- Sensor independence
- Easy scalability
- Robust tracking

---

# 🎯 Data Association

Detected objects from consecutive frames are matched using two complementary techniques.

## Mahalanobis Distance

Used for motion consistency.

It measures how likely a new observation belongs to an existing tracked object while accounting for uncertainty in the EKF covariance matrix.

Benefits:

- Handles uncertainty
- Motion-aware matching
- Reduces incorrect associations

---

## Appearance Matching

To prevent identity switches, the RGB appearance of vehicles is compared using normalized color histograms.

Similarity is computed using Cosine Similarity.

Benefits:

- Prevents identity swaps
- Robust during close interactions
- Improves long-term tracking

---

# 📈 Extended Kalman Filter (EKF)

Each tracked object is represented using the state vector:

```
x =
[
px
py
velocity
yaw
yaw_rate
]
```

The EKF performs:

- State prediction
- Covariance prediction
- Measurement update
- State correction

Advantages:

- Smooth trajectories
- Noise reduction
- Robust tracking
- Velocity estimation

---

# 📊 Tracking Pipeline

The complete tracking loop consists of:

1. Receive synchronized sensor data
2. Detect vehicles
3. Extract object features
4. Associate detections
5. Predict EKF states
6. Update matched tracks
7. Create new tracks
8. Remove stale tracks
9. Visualize tracking
10. Evaluate against ground truth

---

# 📈 Performance Evaluation

When the simulation ends, tracked trajectories are compared against CARLA's ground truth annotations.

The following metrics are computed.

## Precision

Percentage of detected objects that are correct.

---

## Recall

Percentage of actual vehicles successfully detected.

---

## F1 Score

Harmonic mean of Precision and Recall.

---

## Average Translation Error (ATE)

Average localization error between predicted and ground-truth positions.

Lower values indicate better localization.

---

## Bird's-Eye View IoU

Measures overlap between predicted and actual bounding boxes.

Higher values indicate better localization.

---

## ID Switches

Counts how often tracked identities change incorrectly.

Lower values indicate better tracking consistency.
